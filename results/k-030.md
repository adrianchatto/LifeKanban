# k-030 — More Kanban enhancements

**Done by the Kanban worker, 08/06/2026.** Three things were asked for: attachments/screenshots on cards, an AI-built title separate from the description, and being asked for assignee, close date and priority when adding a card. Here is what I built and what changed.

## 1. Attachments and screenshots on cards — built into the app

The New/Edit card modal now has an **Attachments** section. You can add files three ways:

- **Drag a file** onto the drop zone
- **Choose file** button (multiple selection allowed)
- **Paste a screenshot** — copy any image (Cmd-Shift-Ctrl-4 to clipboard, or copy from anywhere) and paste while the card modal is open

Image attachments show a thumbnail; other files show a 📄 icon with a clickable link. Cards on the board show a **📎 N** badge when they have attachments. Each attachment is stored on the card as `{name, url, orig, type, size}`.

**How it works under the bonnet:**

- `server.py` gained `POST /api/upload` (raw body, `X-Filename` header) and `GET /attachments/<file>`. Uploads are sanitised, given a unique timestamped name, and written to a new `attachments/` folder. Path-traversal is blocked and there's a 25 MB size cap (`KANBAN_MAX_UPLOAD`).
- `index.html` gained the upload UI, paste/drag/drop handlers, thumbnail rendering, the card badge, and persistence of the `attachments` array on save.
- Files are kept **local** — `attachments/` is now in `.gitignore`, so screenshots are **not** pushed to the public GitHub repo. Card metadata (the links) still syncs; the image bytes stay on your Mac. If you'd rather have attachments synced too, remove that one line from `.gitignore`.

**You'll need to restart the board server** for the new endpoints to load (Start Kanban.command, or stop/restart the running server). The front-end change is live on next page refresh; the upload won't work until the server is restarted.

Tested end-to-end in a throwaway instance: upload returns a URL, the file serves back byte-identical, oversize is rejected, and traversal attempts return 403.

## 2 & 3. Title-building and the assignee/date/priority prompts — these are chat behaviours, not buttons

These two asks are about how I create a card when you say "add this to my kanban" in chat, so they can't be "built" as app features — they're rules for me. I've written them into the canonical place every session reads: `skills/kanban/SKILL.md`, under a new **"Creating a card from a chat request"** section. From now on, when you ask me to add a card in chat I will:

1. **Write a short title myself** (≈3–8 words) rather than dropping your whole sentence in — e.g. "Chase Acme for signed SOW".
2. **Put the detail in the description.**
3. **Ask you, in one quick prompt, for assignee (Me/Claude), a close date (or none), and a priority (high/medium/low/none)** before adding — skipping anything you've already told me.

This takes effect for future chat sessions because the rule now lives in the skill file, which is loaded whenever the board is used.

## Files changed

- `server.py` — upload + attachment-serving endpoints, image MIME types, `attachments/` dir
- `index.html` — attachments UI, paste/drag/drop, thumbnails, card badge, save logic
- `skills/kanban/SKILL.md` — chat-add workflow (title/assignee/date/priority) + attachments note
- `.gitignore` — exclude `attachments/`

## One thing to decide

I gitignored `attachments/` so screenshots don't end up on the public GitHub repo — the safer default. The trade-off: if you ever restore the board on another machine from GitHub, attachment thumbnails won't be there (the links will 404). If you'd prefer attachments travel with the board, say so and I'll remove that line. Flagging it rather than silently choosing for you.
