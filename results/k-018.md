# k-018 ‚Äî Run the Kanban app in Docker

**Done.** The board can now run as a Docker container, with a `docker-compose.yml` in the repo. Data persists in a volume across rebuilds, and the existing desktop/worker setup is completely unaffected.

## How to run it

From the repo folder:

```
docker compose up -d --build
```

Then open **http://localhost:8787**. To stop it: `docker compose down` (the board data stays in the volume).

## What was added

- **`Dockerfile`** ‚Äî based on `python:3.12-slim`. The app uses only the Python standard library, so there are no dependencies to install and the image stays small. It copies the app code and static assets, bakes in the current `board.json` and `results/` as first-run seed data, exposes port 8787, and includes a `HEALTHCHECK` that pings the board API.
- **`docker-compose.yml`** ‚Äî builds the image, maps `8787:8787`, sets the container environment, mounts a named volume `kanban-data` at `/data`, and restarts unless stopped. A comment explains how to swap the named volume for a bind mount if you'd rather keep data in the repo folder.
- **`docker-entrypoint.sh`** ‚Äî on first run (empty volume) it seeds `board.json` and the `results/` directory into `/data`, then starts the server headless (`--no-browser`). On later runs it leaves your existing data untouched.
- **`.dockerignore`** ‚Äî keeps the build context lean (excludes `.git`, the macOS `.app`/`.command`/`.plist` launchers, lock files, caches). `board.json` and `results/` are deliberately kept so they can seed the volume.

## Code changes (backwards-compatible)

Two small, safe changes let the same code run both on the desktop and in a container:

- **`server.py`** ‚Äî the data location and bind address are now read from environment variables: `KANBAN_DATA` (where `board.json` and `results/` live) and `KANBAN_HOST` (the bind address). Both **default to today's behaviour** ‚Äî the app directory and `127.0.0.1` ‚Äî so nothing changes when you run it directly on your Mac. In the container they're set to `/data` and `0.0.0.0` (binding to all interfaces is required for Docker's port mapping to reach the server).
- **`kanban.py`** ‚Äî the same `KANBAN_DATA` override, so the CLI (and the Claude worker) read and write the same `board.json` as the server when a data dir is configured. Default unchanged.

## Important notes

- **The background Claude worker is separate.** This container runs the *web app* (the board UI and API). The scheduled worker that processes "Claude" cards runs on your Mac as before. If you ever want the worker to run against the containerised data too, point it at the same data dir with `KANBAN_DATA` ‚Äî but that's not needed for normal use.
- **Your data is safe on rebuild.** Because data lives in the `kanban-data` volume, `docker compose up --build` keeps your cards. The seed copy in the image is only used to populate an *empty* volume on the very first run.
- **GitHub sync** (`git_sync.sh`) is a desktop helper and isn't part of the container; it's excluded from the image.
