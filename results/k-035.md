# k-035 — GitHub not updating: diagnosis

Your instinct was right. The board has **not** been reaching GitHub. The last commit
that actually made it to the remote was at **22:20 last night (07/06)**. Everything since
then is stuck locally. Here's exactly why, and what's needed to fix it.

## What I found

**1. There's an unpushed commit, and a stale lock blocking new ones.**
Local `main` is one commit ahead of `origin/main`. A commit was made at **09:08 today**
but never pushed, and the process that made it appears to have crashed — it left a stale
`.git/index.lock` (empty, timestamped 09:08) behind. That lock blocks any *new* commit.

**2. The worker sandbox has no GitHub credentials.**
The scheduled worker runs in a Linux sandbox, not on your Mac. From there:

- It can *reach* GitHub over HTTPS, but there are **no stored credentials** — no token,
  no credential helper, no SSH key. A push gets as far as asking for a username and then
  fails (`could not read Username for 'https://github.com'`).
- Your git credentials live in your Mac's keychain. They are not available in the
  sandbox, by design.

So a push can only succeed when the sync runs **on your Mac**.

**3. The sync script was hardcoded to the Mac path.**
`git_sync.sh` started with `cd /Users/adrianchatto/Documents/Claude/Projects/Kanban`.
That path doesn't exist in the sandbox (the repo is mounted elsewhere), so when the
worker tried to run it, the script aborted on line 1 — before it could commit or push.
That's why you saw no `.sync-result.txt` activity and no pushes from the automated runs.

Net effect: automated runs couldn't push (no creds, wrong path), and the one local
commit that exists is wedged behind a stale lock. Hence ~13 hours of silence on GitHub.

## What I fixed

I rewrote `git_sync.sh` so it is **path-independent** — it now locates its own directory
(`cd "$(dirname "$0")"`), so it runs correctly both on your Mac and in the sandbox. I
also made it clear a *stale* lock automatically (only if the lock file is empty and older
than 2 minutes, so it never disturbs a live commit), and report each step to
`.sync-result.txt`.

I could not commit or push that fix from the worker, for the same two reasons above
(stale lock + no credentials). The edited file is saved on disk, so it'll be included in
the next successful sync.

That will push the backlog (the 09:08 commit, my k-033/k-034/k-035 work, and the script

## To stop this recurring

The real issue is that the **automated worker can't push** — it has no GitHub
credentials in its sandbox. Two ways to fix it properly, pick one:

- **Make the worker run the sync on your Mac** (rather than in the sandbox) via the
  osascript/"Control your Mac" path described in the kanban SKILL's "Method B". That uses
  your Mac credentials and works today. (Best option — no secrets to manage.)
- **Give the sandbox a credential** — e.g. a fine-grained GitHub Personal Access Token in
  a credential helper or `~/.git-credentials`. This works unattended but means storing a
  token; rotate/scope it tightly if you go this way.
