# k-020 — Add a chat bot to the Kanban

**Done.** You can now add cards by typing plain English into a chat panel on the board. The feature is built into the board's web app (`index.html`), so it works wherever you already open the board — no extra install, no setup.

## What was added

A floating **"✦ Add by chat"** button sits in the bottom-right of the board. Click it and a chat panel opens. Type a task the way you'd say it, the assistant shows you the card it understood, and you confirm with **Add card** (or **Edit…** to fine-tune, or **Cancel**). The card is written straight into the board through the same path the "+ New card" button uses, so it shows up instantly and syncs like any other card.

It also answers a few quick questions without leaving the panel — "what's on my board", "what's due today", "what's overdue", "what's assigned to Claude".

## How it reads your message

The assistant parses your sentence locally in the browser. It picks out:

- **Due date** — `today`, `tonight`, `tomorrow`, a weekday (`Friday`, `next Monday`), `this weekend`, `in 3 days`, `in 2 weeks`, `next week`, or an explicit date in UK format (`12/06/2026`, `15 June`, `June 5 2027`, `2026-07-01`).
- **Project** — either a known project name mentioned in the sentence (e.g. `NEO-015`, `Home`) or a `#tag` (e.g. `#Cloudinteract`). Defaults to **General**. A `#tag` for a new name creates that project automatically.
- **Who it's for** — defaults to **you (Ch@o)**. Say **"Claude"** ("have Claude…", "ask Claude to…") to assign it to Claude, and the background worker will pick it up.
- **Repeats** — `daily` / `every day`, `every weekday`, `weekly` / `every Monday`, `monthly`. Recurring cards drop into the Calendar view, matching the existing behaviour.

The rest of the sentence becomes the card title, with filler stripped ("remind me to", "add a task", "todo:", trailing "for/by/on", etc.) and the first letter capitalised.

## Examples (what it produces)

| You type | Title | Project | Owner | Due | Repeats |
|---|---|---|---|---|---|
| Pay parking tomorrow | Pay parking | General | Me | next day | — |
| Review PR for NEO-015 by 12/06/2026 | Review PR | NEO-015 | Me | 12 Jun 2026 | — |
| Have Claude draft the status report for NEO-015 due tomorrow | Draft the status report | NEO-015 | Claude | next day | — |
| Submit timesheet every Friday | Submit timesheet | General | Me | — | weekly (Fri) |
| Standup notes every weekday for Cloudinteract | Standup notes | Cloudinteract | Me | — | every weekday |
| Buy milk #Home | Buy milk | Home | Me | — | — |
| Renew passport by June 5 2027 | Renew passport | General | Me | 5 Jun 2027 | — |

The conversation is remembered between visits (stored in the browser only), and there's a built-in **"help"** command listing what it understands.

## Design decision worth flagging

The card asked to "chat to Claude". I built the parser to run **entirely in the browser** rather than calling the Claude API, for three reasons:

1. **No secrets, no setup.** A live Claude call from the board would need an API key sitting on your machine and a network round-trip on every message. The local parser needs neither and works offline.
2. **It's deterministic and fast.** For the actual job — turning a sentence into title/project/owner/date/repeat — pattern matching is reliable and instant, and you always see and confirm the card before it's added, so there are no surprises.
3. **Privacy.** Your task text never leaves the machine.

If you'd later want true conversational Claude (free-form back-and-forth, fuzzier phrasing, follow-up questions), that's a clean next step: add a small `/api/chat` endpoint to `server.py` that proxies to the Anthropic API using a key from an environment variable, and have the panel fall back to the local parser when no key is set. I've kept the front-end structured so that swap is straightforward. Say the word and I'll wire it up — it would just need you to provide the API key.

## Files changed

- `index.html` — added a self-contained chat module (CSS, the parser, and the panel UI) appended after the existing script. Nothing in the existing board logic was modified; the module reuses the app's own `render`, `pushBoard` and card-creation flow.

Tested: the parser was run against 18 varied phrasings (dates, UK date formats, projects, `#tags`, Claude assignment, all four repeat types) and the page was served locally to confirm the widget loads and the board API still returns cleanly.
