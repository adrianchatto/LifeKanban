# Multi-user support for LifeKanban — research and options

**Card:** k-036 · Create users
**Date:** 08/06/2026
**Status:** Research / options for your decision. No code changed.

---

## What you asked for

The ability to add and manage users on the Kanban board so each person gets
their own interface showing just their tasks, each with their own API key — and
this has to keep working once the board moves to Docker. You asked me to research
the options first and then lay them out.

## Where the board is today (the constraints that shape every option)

I read the actual code before writing this, because the right answer depends
heavily on how the thing is built. The relevant facts:

- **Zero dependencies.** `server.py` is pure Python standard library — no Flask,
  no auth library, nothing from pip. That is a deliberate design choice and the
  reason the Docker image is tiny. Any option that drags in a web framework or an
  identity provider throws that away.
- **No authentication at all right now.** Anyone who can reach the port can read
  and overwrite the whole board. On `127.0.0.1` (desktop) that is fine — only you
  can reach it. The moment it runs in Docker bound to `0.0.0.0` and is reachable
  from your network or the internet, that is a wide-open door.
- **The data model already has the hook.** Every card carries an `assignee`
  field (`Ch@o` or `Claude` today). Per-user *views* are therefore mostly a
  filtering and identity problem, not a data-modelling one — you don't need to
  restructure `board.json` to slice it by person.
- **The write path is coarse.** `POST /api/board` overwrites the *entire* board
  in one shot; the frontend holds the whole board in memory and re-saves all of
  it. That matters: if two users edit at once, last-write-wins clobbers the
  other's changes, and a per-user write path would need finer-grained endpoints
  or optimistic concurrency.
- **The worker and the CLI assume one board.** `kanban.py claim-next`, the
  scheduled worker, and the GitHub sync all operate on a single `board.json`.
  Whatever we do should not break the Claude worker or the sync.

So the real question is not "can we add users" — it's *how much* identity and
isolation you actually want, traded against keeping this simple and dependency-free.

## Three things people mean by "users" — worth separating

1. **Authentication** — proving who you are (login / API key).
2. **Authorisation / isolation** — what you're allowed to see and change (your
   cards vs everyone's).
3. **Personalised view** — the UI showing "my tasks" by default.

You can have (3) with neither (1) nor (2) — that's just a filter. A real
multi-user system needs all three. The options below climb that ladder.

---

## Option A — Per-user views only (no real auth)

**What it is:** Add a `users` list to `board.json` and a `?user=Ch@o` URL param
(or a name picker in the top bar). The UI filters cards to that assignee and
remembers the choice in `localStorage`. No passwords, no API keys, no isolation —
anyone can switch to anyone.

**Effort:** Half a day. Frontend-only, plus a small `users` array in the board.

**Pros:** Trivial. Keeps zero dependencies. Gives you the "their own interface
with just their tasks" feel immediately. No risk to the worker or sync.

**Cons:** Not security. Anyone on the network sees and edits everything. No API
keys, so it doesn't satisfy that part of your ask. Only sensible if every "user"
is trusted and the board stays on your machine or a private LAN.

**Verdict:** A good *first step* or a fallback, but on its own it doesn't meet the
brief (you explicitly want API keys and a move to Docker, i.e. off `localhost`).

---

## Option B — API-key auth with per-user scoping (recommended)

**What it is:** Add a `users` array to `board.json`, each entry holding a name,
a role, and a hashed API token:

```json
"users": [
  {"id": "u1", "name": "Ch@o", "role": "admin",
   "token_sha256": "…", "created": "2026-06-08"}
]
```

- The server requires an `Authorization: Bearer <token>` header (or a
  `?key=` param for the browser, stored in a cookie/`localStorage`) on every
  `/api/*` call. No valid token → `401`.
- We store only the **SHA-256 hash** of each token, never the token itself.
  The plaintext key is shown **once** at creation, then it's gone — same model
  as GitHub personal access tokens.
- The server filters the board to the caller's assignee on read, and rejects
  writes to cards that aren't theirs — except `admin` (you), who sees and edits
  everything. That keeps your "I still have full access in Docker" requirement.
- A tiny CLI surface manages users without hand-editing JSON:
  `kanban.py user-add "Sam" --role member`,
  `kanban.py user-token <id> --rotate`, `kanban.py user-list`,
  `kanban.py user-rm <id>`.
- The Claude worker gets its own admin token via an env var so the scheduled
  loop keeps working.

**Effort:** 1–2 days. All achievable with the standard library — `hashlib` and
`secrets` are built in, so **still zero pip dependencies**. The work is: a
`users` block, an auth check in the request handler, assignee-scoped
read/write filtering, the CLI commands, and a login box in the UI.

**Pros:** Meets the brief — real per-user API keys, per-user views, you retain
admin access, works the same on `localhost` and in Docker. Keeps the project's
no-dependency ethos. Tokens are hashed at rest. Granular enough to be genuinely
useful without being heavy.

**Cons:** API keys are bearer tokens — whoever holds one *is* that user, so they
need to be sent over HTTPS once you expose the board beyond your LAN (see the
deployment note below). No "forgot password" flow, but rotation via the CLI
covers that. The coarse `POST /api/board` write path needs splitting into
per-card or per-user writes to enforce isolation properly — that's the bulk of
the effort.

**Verdict:** The right fit. It's the lightest thing that actually does what you
asked, and it doesn't betray the design.

---

## Option C — Full username/password auth with sessions

**What it is:** Proper login — username + password (hashed with a slow KDF),
server-issued session cookies, optional "remember me", logout, password change.
API keys become an *additional* per-user credential for scripts/the worker, on
top of interactive login.

**Effort:** 3–5 days. Sessions, CSRF protection, password hashing
(`hashlib.scrypt` is in the stdlib, so still no pip), login/logout pages, cookie
handling. Meaningfully more surface area to get right and to keep secure.

**Pros:** The most familiar UX (a normal login screen). Best if non-technical
people will use it, or if you'll have more than a handful of users.

**Cons:** Materially more code and more ways to introduce a security bug
(session fixation, CSRF, cookie flags) in what is currently a ~300-line server.
Overkill for a personal/small-team board. Doesn't add much over Option B unless
you specifically need interactive password login rather than keys.

**Verdict:** Only worth it if this grows into something several non-technical
people log into daily. For now it's more than you need.

---

## Option D — Put an auth proxy in front (don't build auth at all)

**What it is:** Leave `server.py` untouched and stand a reverse proxy in front of
it in the Docker setup — e.g. Caddy or Nginx with basic-auth, or an
authenticating gateway like Authelia / oauth2-proxy. The proxy handles login and
TLS; the Kanban app stays dumb.

**Effort:** Half a day to a day of *config* (no app code), but it adds a second
container and moving parts to the compose file.

**Pros:** Zero changes to the Python. Battle-tested auth and HTTPS handled by
mature software. Good if you just want to lock the whole board behind one
gate when it goes on a server.

**Cons:** It's all-or-nothing — the proxy can keep strangers out, but it
**cannot give per-user views or per-user API keys**, because it doesn't
understand the board's `assignee` field. So it solves "keep the board private in
Docker" but **not** "each user sees only their tasks with their own key." You'd
likely end up pairing it with Option A or B anyway.

**Verdict:** Worth combining with B for the TLS/edge layer, but not a substitute
for in-app users.

---

## Recommendation

**Go with Option B (API-key auth with per-user scoping), and use a reverse proxy
for TLS only when you expose it publicly.** It's the smallest change that
actually delivers what you asked — per-user interfaces, per-user API keys, your
own continued admin access, and safe operation in Docker — without abandoning the
zero-dependency design that makes this project nice to run.

If you want to dip a toe in first, **ship Option A this week** (per-user view
filtering) and layer Option B's auth on top once you're happy with the view
behaviour. A is a strict subset of B, so nothing is wasted.

## Security notes for the Docker move (important either way)

- **Don't expose `0.0.0.0` without auth.** Today the board has no login. The
  instant it's reachable beyond `localhost`, anyone who finds the port owns the
  board. Auth (Option B/C) or a gating proxy (D) must land *before* it's exposed.
- **Use HTTPS once it leaves your machine.** Bearer tokens and passwords over
  plain HTTP can be sniffed. Terminate TLS at a proxy (Caddy gets you automatic
  Let's Encrypt certs with near-zero config) or behind a VPN/Tailscale.
- **Keep secrets out of GitHub.** `board.json` syncs to a public-ish repo. If we
  add users, we store only *hashed* tokens, never plaintext — and it's worth
  double-checking `.gitignore` so nothing with live credentials is ever pushed.
- **Give the Claude worker its own token** (env var, admin role) so the scheduled
  loop and `claim-next` keep working after auth is switched on.
