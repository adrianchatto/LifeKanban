# k-044 — Why Claude cards get stuck in "Doing", and the fix

## The cause

The worker only has one way in: `kanban.py claim-next`. That command selects
**only** cards with `status == "todo"`, moves the oldest to `doing`, and then the
same worker pass is expected to do the work and move it on to `done` or
`needs_ok`.

The problem is what happens when a pass doesn't finish. If the scheduled worker
session times out, hits a tool error, or crashes after `claim-next` but before it
moves the card on, the card is left sitting in `doing`. The next run calls
`claim-next` again — which ignores everything already in `doing` — so the
stranded card is **never retried**. There was no reaper, timeout, or recovery
path anywhere in the code. One failed pass orphaned a card permanently, and over
time `doing` silently accumulated dead cards.

This is a structural gap, not a one-off glitch: any interrupted worker run
produces a stuck card.

## The fix

`claim-next` is now self-healing. Before it claims new work, it runs a recovery
sweep (`requeue_stale`) that:

- Finds **Claude** cards sitting in `doing` whose `updated` stamp is older than
  `STALE_DOING_MIN` (default **30 minutes**; override with the `KANBAN_STALE_MIN`
  env var) and sends them back to `todo` so the next claim re-attempts them. The
  `updated` stamp moves forward whenever the worker logs progress, so a card that
  is genuinely being worked on is never yanked away mid-task.
- Guards against a "poison" card that keeps failing: after `MAX_REQUEUES`
  bounces (default **3**), the card is parked in `needs_ok` with a note rather
  than looping forever and eating worker passes.
- Touches **only** Claude-assigned cards. Cards you've moved into `doing`
  yourself (e.g. k-010, "Port Support-Shere to Azure") are left exactly where
  they are.

A successful `move <id> done` clears the recovery counter, so a card rescued once
isn't penalised if it's ever reworked later.

There's also a manual command for diagnostics or an immediate cleanup:

```
python3 kanban.py requeue-stale
```

It runs the same sweep on demand and prints what it changed.

## What this means in practice

Recovery is automatic — the existing ~15-minute worker schedule already calls
`claim-next`, so no new scheduling is needed. A worker that dies mid-card now
self-corrects on its next pass: the stranded card returns to `todo` and gets
picked up again, with a clear audit trail in the card log.

## Verified

Tested on an isolated copy of the board:

- Stale Claude `doing` card → requeued to `todo` ✓
- Card updated 5 min ago (actively working) → left in `doing` ✓
- Human (Ch@o) card stuck in `doing` → untouched ✓
- Card already requeued 3× → parked in `needs_ok` ✓
- `claim-next` recovers a stale card **and** claims it in the same pass ✓
- `move … done` clears the requeue counter ✓

## One thing to note

k-010 ("Port Support-Shere to Azure") is sitting in `doing` but it's **your**
card, moved there manually — not a worker orphan. The fix deliberately leaves it
alone. If it's stalled for a different reason, that's a separate conversation.
